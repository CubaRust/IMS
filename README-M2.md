## 编写新测试

### 1. 业务层集成测试(推荐)

在 `crates/cuba-e2e/tests/` 下加 `e2e_<xxx>.rs`,用 `TestDb` + fixtures:

```rust
use cuba_testkit::{TestDb, fixtures::*};
use cuba_<module>::<Service>;

#[tokio::test]
async fn my_scenario() {
    let db = TestDb::new().await;
    let pool = db.pool_owned();
    let ctx = admin_ctx();

    let svc = <Service>::new(pool.clone());
    // ... 调 svc 的方法,断言 DB 状态
}
```

`fixtures` 里已有:`admin_ctx / dec / today / seed_material / seed_supplier /
seed_customer / default_raw_wh_loc / default_bad_wh_loc / default_scrap_wh_loc`。

### 2. HTTP 层测试

```rust
use cuba_testkit::{TestDb, http::spawn_server};

#[tokio::test]
async fn http_case() {
    let db = TestDb::new().await;
    let (base, _h) = spawn_server(db.pool_owned()).await;
    let client = reqwest::Client::new();

    let r = client.post(format!("{base}/api/v1/..."))
        .bearer_auth(&token)
        .json(&json!({...}))
        .send().await.unwrap();
}
```

### 3. 纯单元测试(不需 DB)

放在业务 crate 自己的 `src/**/mod.rs` 里,用 `#[cfg(test)] mod tests`。
例如 `cuba-inventory/src/domain/rules.rs` 有 7 条 rules 单测。

## 加新业务端点的套路

假设要加"批次合并"接口:

1. `crates/cuba-<module>/src/application/mod.rs`
   - 定义 `CreateMergeCommand` + Query + View + `Service::merge()`
2. `crates/cuba-<module>/src/infrastructure/repository.rs`
   - trait 方法 + PG 实现
3. `crates/cuba-api/src/router/<module>.rs`
   - 加 `async fn merge_batch(...)` handler
   - 在 `routes()` 里 `.route("/.../merge", post(merge_batch))`
4. 加测试:`crates/cuba-e2e/tests/e2e_<module>.rs`
5. (可选)在 `crates/cuba-api/src/openapi.rs` 加 `#[utoipa::path]` 示例

**不要**在业务 crate 里 import `axum`,HTTP 相关一律在 `cuba-api`。

## 调 OpenAPI

- `GET /docs` — Swagger UI
- `GET /api-docs/openapi.json` — 原始 OpenAPI 3.1 JSON

目前骨架有 4 个示例端点注解(login / balance / inbound-create / inbound-submit)。
要加新端点注解,在 `crates/cuba-api/src/openapi.rs` 里:

1. 写 `#[utoipa::path(...)] fn _doc_xxx() {}`(dummy 函数,只为装饰)
2. 在 `ApiDoc` 的 `paths(...)` 列表里注册 `_doc_xxx`
3. 若需要新 schema,在 `openapi_schema.rs` 加一个 `ToSchema` 结构体并注册到 `components(schemas(...))`

## DB 变更流程

1. 在 `migrations/` 下加下一个递增编号的 SQL 文件(当前最大 0015)
2. `just db-reset`(或 `just dev-purge && just dev`)重跑所有 migration 验证
3. 若要改现有表,**不要修改老 migration**,而是新写一个 alter 文件
4. 加 index 建议加 `where <filter>` 部分索引降低开销
5. 所有表 `id bigint generated by default as identity` 主键 + `created_at/updated_at`(auto touch trigger 已由 0001 的 `sys.touch_updated_at()` 建好)

## CI 说明

`.github/workflows/ci.yml` 4 个 job:

| job | 作用 | 依赖 Docker |
|---|---|---|
| check | fmt check + clippy -D warnings + build | 否 |
| unit-tests | `cargo test --exclude cuba-e2e --lib` | 否 |
| e2e-tests | testcontainers 起 pg,跑 e2e | **是**(runner 自带) |
| migrations | service container pg,顺序执行所有迁移 + verify.sql | **是** |

所有 PR 必须 4 个 job 全绿才能 merge。

## 常见问题

**Q: e2e 在本地卡住不动 ?**
A: 检查 Docker 是否在跑。testcontainers 会拉 `postgres:16`(首次 ~150 MB)。

**Q: clippy 报依赖版本冲突 ?**
A: `cargo update -w` 刷 lockfile。若仍有,先 `rm Cargo.lock && cargo build` 重建。

**Q: 本地 migrations 跑飘了 ?**
A: `just dev-purge && just dev` 一键重置。

**Q: 单元测试不需要 docker 对吧 ?**
A: `just unit` 排除了 `cuba-e2e`,只跑 crate 内的 `#[test]`,不启 container。

**Q: 怎么看业务错误码 ?**
A: 错误码分段在 [`README-M1.md`](./README-M1.md#核心约束不可绕过) 的第 3 条。
`cuba-shared/src/error` 下 `ErrorCode` 定义了所有常量。
